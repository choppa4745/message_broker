# Как протестировать practice4 (аномалии изоляции)

Этот файл — пошаговая инструкция: от поднятия БД до проверки каждой аномалии и критериев «тест пройден».

---

## 1. Что понадобится

- **Docker** с запущенным демоном (на macOS обычно Docker Desktop).
- Клиент **`psql`** в `PATH` (часто ставится вместе с PostgreSQL; можно вызвать из контейнера, см. §8).
- Два **терминала** одновременно (или два вкладки IDE) — для «Сессии A» и «Сессии B».

Рекомендуется задать переменную подключения во всех терминалах:

```bash
export PGURL="postgresql://lab_user:lab_pass@localhost:5441/isolation_lab"
```

Строка должна совпадать с тем, что указано в `docker-compose.yml`: порт хоста **5441**, пользователь `lab_user`, пароль `lab_pass`, база `isolation_lab`.

---

## 2. Запуск и остановка контейнера

Из корня каталога проекта перейдите в `practice4` и выполните:

```bash
cd practice4
docker compose up -d
```

Проверка, что контейнер жив:

```bash
docker compose ps
```

В колонке статуса должно быть что-то вроде `running` и `healthy` (если healthcheck успел отработать).

Остановка после тестов:

```bash
docker compose down
```

**Если контейнер не стартует:** убедитесь, что Docker запущен; порт 5441 не занят другим процессом (`lsof -i :5441` на macOS/Linux).

---

## 3. Полный сброс схемы и начальные данные

Выполняйте **до** каждого блока демо, если хотите предсказуемые цифры, или после «испорченного» состояния.

```bash
psql "$PGURL" -f sql/00_reset.sql
psql "$PGURL" -f sql/01_schema_and_seed.sql
```

**Контроль после сида** (третье подключение или отдельная сессия после `COMMIT`):

```sql
SELECT id, name, balance FROM accounts ORDER BY id;
SELECT id, account_id, amount FROM orders ORDER BY id;
```

Ожидается: у Alice `id = 1`, `balance = 1000.00`; минимум один заказ у `account_id = 1`.

---

## 4. Общие правила параллельного тестирования

1. Откройте **два** интерактивных `psql`:

   ```bash
   psql "$PGURL"
   ```

   В одном работайте как **Сессия A**, в другом — **Сессия B**.

2. Команды выполняйте **в указанном порядке**. Номера шагов в разделах ниже этим и обусловлены.

3. Если в инструкции сказано «не коммитьте» — не выполняйте `COMMIT`/`ROLLBACK` для этой транзакции до следующего шага.

4. После демо закрывайте «висящие» транзакции (`COMMIT` или `ROLLBACK`), иначе блокировки помешают следующим тестам.

5. Для отчёта полезно сохранить **скриншоты** обоих терминалов с текстом команд и выводом, плюс финальный `SELECT` по таблицам.

Подробные тексты сценариев дублируются в файлах:

- `sql/02_non_repeatable_read.sql`
- `sql/03_phantom_read.sql`
- `sql/04_lost_update.sql`
- `sql/05_dirty_read_postgresql_note.sql`

## 5. Тест: non-repeatable read (неповторяемое чтение)

**Цель:** в одной транзакции два чтения одной и той же строки дают **разные** значения после коммита другой сессии (уровень `READ COMMITTED`).

### 5.1 Подготовка

```bash
psql "$PGURL" -f sql/00_reset.sql
psql "$PGURL" -f sql/01_schema_and_seed.sql
```

### 5.2 Шаги

| Шаг | Где | Что выполнить |
|-----|-----|----------------|
| 1 | Сессия A | `BEGIN;` |
| 2 | Сессия A | `SET TRANSACTION ISOLATION LEVEL READ COMMITTED;` |
| 3 | Сессия A | `SELECT balance FROM accounts WHERE id = 1;` — зафиксируйте значение (обычно `1000.00`). **Не коммитьте A.** |
| 4 | Сессия B | `BEGIN;` |
| 5 | Сессия B | `UPDATE accounts SET balance = balance - 100 WHERE id = 1;` |
| 6 | Сессия B | `SELECT balance FROM accounts WHERE id = 1;` — ожидается `900.00` |
| 7 | Сессия B | `COMMIT;` |
| 8 | Сессия A | `SELECT balance FROM accounts WHERE id = 1;` — ожидается **уже `900.00`**, тогда как первый SELECT в A был `1000.00` |
| 9 | Сессия A | `COMMIT;` |

### 5.3 Критерий успеха

Внутри **одной** транзакции A первое и второе `SELECT balance ... id = 1` **различаются** после шага 7.

### 5.4 Если что-то пошло не так

- Если шаг 8 показывает всё ещё `1000.00` — проверьте, что B действительно сделал `COMMIT` до второго SELECT в A и что в A не выставлен `REPEATABLE READ`.

## 6. Тест: phantom read (фантомное чтение)

**Цель:** в одной транзакции два одинаковых по смыслу запроса с условием дают **разное количество строк** после вставки и коммита другой сессии.

### 6.1 Подготовка

```bash
psql "$PGURL" -f sql/01_schema_and_seed.sql
```

(Если таблицы уже существуют с «мусором», сначала `00_reset.sql`, затем снова `01_schema_and_seed.sql`.)

### 6.2 Шаги (READ COMMITTED)

| Шаг | Где | Что выполнить |
|-----|-----|----------------|
| 1 | Сессия A | `BEGIN;` |
| 2 | Сессия A | `SET TRANSACTION ISOLATION LEVEL READ COMMITTED;` |
| 3 | Сессия A | `SELECT COUNT(*) FROM orders WHERE account_id = 1;` — после сида обычно **1**. **Не коммитьте A.** |
| 4 | Сессия B | `BEGIN;` |
| 5 | Сессия B | `INSERT INTO orders (account_id, amount) VALUES (1, 25.00);` |
| 6 | Сессия B | `COMMIT;` |
| 7 | Сессия A | `SELECT COUNT(*) FROM orders WHERE account_id = 1;` — ожидается **2** |
| 8 | Сессия A | `COMMIT;` |

### 6.3 Критерий успеха

Числа из шагов 3 и 7 **различны** при одной транзакции A.

### 6.4 Дополнительный тест (сравнение с REPEATABLE READ)

Восстановите данные (`01_schema_and_seed.sql`), повторите шаги 1–8, но в шаге 2 вместо `READ COMMITTED` выполните:

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

Ожидается: шаг 3 и шаг 7 в A дают **одинаковое** значение `COUNT(*)` (виден снимок на начало транзакции; «фантом» внутри A не проявляется).

## 7. Тест: lost update (потерянное обновление)

**Цель:** смоделировать типичную ошибку приложения: прочитали баланс, другой сеанс увеличил баланс и закоммитил, затем первый сеанс записал сумму, посчитанную **от устаревшего чтения**, тем самым затерев вклад второго.

Важно: в PostgreSQL запись вида `UPDATE ... SET balance = balance + 50` **не** демонстрирует ту же логическую ошибку (берётся актуальная версия строки). В сценарии используется **литерал** `1050.00` после чтения «1000 в приложении».

### 7.1 Подготовка

Подключитесь и верните счёт и версию для Alice:

```bash
psql "$PGURL" -f sql/04_lost_update.sql
```

Файл в конце содержит только `UPDATE accounts SET balance = 1000.00, version = 0 WHERE id = 1;` — это сброс перед демо.

### 7.2 Шаги

| Шаг | Где | Что выполнить |
|-----|-----|----------------|
| 1 | Сессия A | `BEGIN;` |
| 2 | Сессия A | `SELECT balance FROM accounts WHERE id = 1;` — ожидается `1000.00`. **Не коммитьте A.** |
| 3 | Сессия B | `BEGIN;` |
| 4 | Сессия B | `SELECT balance FROM accounts WHERE id = 1;` |
| 5 | Сессия B | `UPDATE accounts SET balance = balance + 100 WHERE id = 1;` |
| 6 | Сессия B | `COMMIT;` |
| 7 | Сессия A | `UPDATE accounts SET balance = 1050.00 WHERE id = 1;` — имитация «прочитали 1000, прибавили 50» |
| 8 | Сессия A | `COMMIT;` |
| 9 | Любая сессия | `SELECT balance FROM accounts WHERE id = 1;` |

### 7.3 Критерий успеха

Итоговый баланс **1050.00**. Если бы обе операции применялись к согласованной базе **1000**, ожидали бы **1150** (сначала +100, потом +50 к актуальному); фактически запись A перезаписала результат B — **вклад +100 потерян с точки зрения бизнес-логики**.

## 8. Тест: dirty read и ограничения PostgreSQL

**Цель задания в этом репозитории:** зафиксировать, что классическое **dirty read** (чтение незафиксированных данных другой транзакции **как надёжных**) в PostgreSQL **не воспроизводится**: уровень `READ UNCOMMITTED` работает как не ниже `READ COMMITTED`.

### 8.1 Подготовка

Удобно вернуть данные:

```bash
psql "$PGURL" -f sql/01_schema_and_seed.sql
```

### 8.2 Шаги

| Шаг | Где | Что выполнить |
|-----|-----|----------------|
| 1 | Сессия A | `BEGIN;` |
| 2 | Сессия A | `UPDATE accounts SET balance = 7777.77 WHERE id = 1;` |
| 3 | Сессия A | **Не** делайте `COMMIT`/`ROLLBACK`. |
| 4 | Сессия B | `BEGIN;` |
| 5 | Сессия B | `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;` |
| 6 | Сессия B | `SELECT balance FROM accounts WHERE id = 1;` |

### 8.3 Что считать успешной проверкой

Один из вариантов (зависит от порядка блокировок и версий):

- B **ждёт** завершения транзакции A, и только после `COMMIT`/`ROLLBACK` в A запрос в B возвращает значение; либо
- B не видит закоммиченным «грязное» `7777.77` до явного коммита A.

После демо завершите транзакции, чтобы не блокировать таблицу:

- Сессия A: `ROLLBACK;` или `COMMIT;`
- Сессия B: если транзакция ещё открыта — `COMMIT;` или `ROLLBACK;`

**Не ожидайте** получить в B стабильное отображение незакоммиченных `7777.77` как корректных данных — для отчёта как раз описывают отсутствие dirty read в PG.

## 9. Контрольный чек-лист «вся функциональность проверена»

- [ ] Контейнер поднят, `psql` подключается по `$PGURL`.
- [ ] После `00_reset` + `01_schema_and_seed` видны Alice/Bob и хотя бы один order у Alice.
- [ ] Non-repeatable read: два SELECT в A дали разный `balance`.
- [ ] Phantom read: два `COUNT` в A дали разные числа при `READ COMMITTED`.
- [ ] (Опционально) При `REPEATABLE READ` в A второй `COUNT` совпал с первым.
- [ ] Lost update: после сценария баланс = `1050.00`, не `1150.00`.
- [ ] Dirty read: задокументировано поведение PG (нет чтения незафиксированных версий как валидных данных).
- [ ] После всех экспериментов нигде не осталось открытых транзакций (`COMMIT`/`ROLLBACK`).

## 10. Если нет локального `psql`

Запросы можно выполнять через `exec` в контейнере:

```bash
docker exec -it practice4-postgres psql -U lab_user -d isolation_lab
```

Два таких входа из разных терминалов эквивалентны двум сессиям A и B.
