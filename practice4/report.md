## 2. Non-repeatable read (неповторяемое чтение)

### 2.1 Суть

В одной транзакции два одинаковых по смыслу запроса к одной и той же строке возвращают **разные значения**, потому что другая транзакция **закоммитила** изменение между первым и вторым чтением. При уровне **READ COMMITTED** (по умолчанию в PostgreSQL) для каждого оператора видна «свеже закоммиченная» версия строки.

### 2.2 Шаги воспроизведения

1. Выполнить `00_reset.sql` и `01_schema_and_seed.sql`.
2. Открыть **два** терминала с `psql` к одной БД.
3. Следовать блокам **Сессия A** и **Сессия B** в `sql/02_non_repeatable_read.sql` в указанном порядке.

### 2.3 Результат

*Вставьте скриншот: первый `SELECT` в A, затем коммит в B, второй `SELECT` в A с изменённым `balance`.*

Пример ожидаемого текстового вывода: первое чтение совпадает с начальным сидом (1000.00), после коммита B второе чтение в A показывает новое значение.

### 2.4 Как избежать

- Перевести транзакцию на **REPEATABLE READ** или **SERIALIZABLE** — чтения внутри транзакции согласуются со снимком на момент первого запроса (при RR в PG «неповторяемость» того же значения уходит для обычных SELECT).
- Заблокировать строку при первом чтении: `SELECT ... FOR UPDATE` / `FOR SHARE`, чтобы другая сессия не могла закоммитить конфликтующий UPDATE до завершения A.

## 3. Phantom read (фантомное чтение)

### 3.1 Суть

В одной транзакции два запроса с **одним и тем же условием** (например, `COUNT(*) ... WHERE account_id = 1`) дают **разное количество строк**, потому что другая транзакция **вставила** строки, попадающие под условие, и закоммитила.

### 3.2 Шаги воспроизведения

1. После `01_schema_and_seed.sql` у Alice (`id = 1`) уже есть минимум одна запись в `orders`.
2. В двух сессиях выполнить сценарий из `sql/03_phantom_read.sql`.

### 3.3 Результат

*Скриншот: первый `COUNT(*)` в A, вставка в B, второй `COUNT(*)` в A — число увеличилось.*

Дополнительно (сравнение): повторить с `SET TRANSACTION ISOLATION LEVEL REPEATABLE READ` в начале A — второй `COUNT` в той же транзакции должен **не** измениться (виден снимок на начало транзакции).

### 3.4 Как избежать

- **REPEATABLE READ** / **SERIALIZABLE** в PostgreSQL строятся на снимках и предотвращают «фантомы» в рамках одного снимка (для типичных запросов на чтение).
- В прикладной логике — явные блокировки по диапазону или материализация выборки (временная таблица, курсор с `WITH HOLD` — по необходимости).

## 4. Lost update (потерянное обновление)

### 4.1 Суть

Два сеанса читают одно значение и обновляют строку **как «база + дельта»**, вычисленная от **устаревшего** прочитанного значения. Итоговая запись одного сеанса **перезаписывает** вклад другого так, что суммарный эффект не соответствует последовательному выполнению обеих логик.

### 4.2 Шаги воспроизведения

1. Выполнить сброс из `sql/04_lost_update.sql` (`UPDATE accounts SET balance = 1000...`).
2. Следовать двум сессиям из комментариев в том же файле. В сессии A запись в базу должна имитировать **устаревшее** значение из первого `SELECT` (`UPDATE ... SET balance = 1050.00`), а не `balance = balance + 50` — иначе PostgreSQL возьмёт актуальную строку и **потеря обновления не проявится**.

### 4.3 Результат

*Скриншот: два `SELECT` до/после, финальный `balance` (часто 1050 при последнем коммите A после B с +100 — вклад перезаписан). Опишите численно ожидаемую vs фактическую бизнес-логику.*

### 4.4 Как избежать

- **Пессимистично:** `SELECT ... FOR UPDATE` перед чтением и пересчётом в той же транзакции.
- **Оптимистично:** столбец `version` и `UPDATE ... WHERE id = ? AND version = ?` с проверкой числа затронутых строк.
- **Атомарно:** одна команда `UPDATE accounts SET balance = balance + 50` без предварительного SELECT в приложении.
- **SERIALIZABLE** + обработка ошибок перезапуска транзакции при конфликте сериализации.

## 5. Dirty read (грязное чтение)

### 5.1 Суть (теория)

Чтение **ещё не закоммиченных** изменений другой транзакции. Если та откатится, данные «виденные» читателем оказываются недействительными.

### 5.2 Особенность PostgreSQL

В PostgreSQL уровень **READ UNCOMMITTED** не даёт читать незафиксированные версии строк — фактически используется не ниже **READ COMMITTED**. Незакоммиченные строки при блокировке обновлением **не показываются** как зафиксированные; сессия-читатель может **ждать** снятия блокировки.

### 5.3 Шаги «демонстрации» в PG

Выполнить сценарий из `sql/05_dirty_read_postgresql_note.sql`: незакоммиченный `UPDATE` в A, попытка чтения в B с `READ UNCOMMITTED`.

### 5.4 Результат

*Скриншот: в B не появляется «грязного» значения как закоммиченного; при необходимости — окно ожидания до `COMMIT`/`ROLLBACK` в A.*

**Вывод для отчёта:** на выбранной СУБД классический dirty read **не воспроизведён**; зафиксировано поведение, исключающее такое чтение.

### 5.5 Как избежать (в СУБД, где dirty read возможен)

Использовать уровень не ниже **READ COMMITTED** (как в PostgreSQL по умолчанию).
